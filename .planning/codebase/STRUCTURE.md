## Naming Conventions

**Files:**
- Pages: `page.tsx` (Next.js convention)
- Layouts: `layout.tsx` (Next.js convention)
- Components: `kebab-case.tsx` (e.g., `lead-list.tsx`, `qa-card.tsx`)
- Server Actions: `kebab-case.ts` in `app/actions/` (e.g., `create-lead.ts`)
- Services: `kebab-case.ts` in `lib/services/` (e.g., `lead-service.ts`)
- Utilities: `kebab-case.ts` in `lib/` (e.g., `format-date.ts`)
- Types: `kebab-case.ts` in `types/` or co-located (e.g., `lead-types.ts`)

**Directories:**
- Route segments: `kebab-case` (e.g., `app/openclaw/installatie/`)
- Component folders: `kebab-case` (e.g., `components/qa/`)
- Route groups: `(group-name)` with parentheses (e.g., `app/(dashboard)/leads/`)

**Components:**
- PascalCase for component names (e.g., `LeadList`, `QaCard`, `BlogPost`)
- File name matches component name in kebab-case (e.g., `LeadList` in `lead-list.tsx`)

**Functions:**
- camelCase for functions (e.g., `getLeadById`, `createLead`, `formatDate`)
- Server Actions: verb + noun (e.g., `createLead`, `updateLeadStatus`)

**Constants:**
- UPPER_SNAKE_CASE (e.g., `MAX_LEADS_PER_PAGE`, `DEFAULT_STATUS`)

**Types/Interfaces:**
- PascalCase (e.g., `Lead`, `User`, `CreateLeadInput`)
- Type files: `kebab-case.ts` (e.g., `lead-types.ts`)

## Where to Add New Code

**New Page:**
- Implementation: `app/[route]/page.tsx` (e.g., `app/openclaw/page.tsx`)
- Layout (if needed): `app/[route]/layout.tsx`
- Loading state: `app/[route]/loading.tsx`
- Error boundary: `app/[route]/error.tsx`

**New Component:**
- UI component: `components/ui/[component-name].tsx`
- Feature component: `components/[feature]/[component-name].tsx` (e.g., `components/leads/lead-card.tsx`)
- Co-locate with feature if specific to one area

**New Server Action:**
- Implementation: `app/actions/[feature]/[action-name].ts` (e.g., `app/actions/leads/create-lead.ts`)
- Must include: Auth check, permission check, validation (Zod), service call, error handling

**New Service:**
- Implementation: `lib/services/[entity]-service.ts` (e.g., `lib/services/lead-service.ts`)
- Pattern: Stateless class with async methods
- Must include: Business logic, validation, database operations, audit logging

**New API Route:**
- Implementation: `app/api/[route]/route.ts` (e.g., `app/api/leads/route.ts`)
- Export named functions: `GET`, `POST`, `PATCH`, `DELETE`
- Follow: Auth → Permission → Validate → Execute → Log → Return

**Utilities:**
- Shared helpers: `lib/[utility-name].ts` (e.g., `lib/format-date.ts`)
- Domain-specific utilities: `lib/[domain]/[utility-name].ts`

**Types:**
- Shared types: `types/[entity]-types.ts` (e.g., `types/lead-types.ts`)
- Co-located types: Same directory as component/service
- Database types: Generated by Drizzle from schema

**Database Schema:**
- Schema definition: `packages/database/schema/[entity].ts` (planned)
- Migrations: `packages/database/migrations/[timestamp]_[description].sql` (generated by Drizzle)

**Styles:**
- Global styles: `app/globals.css` (Tailwind utilities, CSS variables)
- Component styles: Use Tailwind classes inline (utility-first approach)
- Custom components: Tailwind config in `tailwind.config.ts`
