  <files>lib/supabase/client.ts, lib/supabase/server.ts</files>
  <action>
Create the `lib/supabase/` directory and two client factory files.

**lib/supabase/client.ts** (Browser client):
- Import `createBrowserClient` from `@supabase/ssr`
- Export a `createClient()` function that calls `createBrowserClient` with `process.env.NEXT_PUBLIC_SUPABASE_URL!` and `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!`
- No custom cookie config needed (browser client handles cookies via document.cookie automatically)
- The function returns a singleton by default (safe to call in every render)
- Use non-null assertions (`!`) -- the app should fail fast if env vars are missing

**lib/supabase/server.ts** (Server client):
- Import `createServerClient` from `@supabase/ssr`
- Import `cookies` from `next/headers`
- Export an `async function createClient()` that:
  1. Calls `const cookieStore = await cookies()`
  2. Returns `createServerClient(url, anonKey, { cookies: { getAll, setAll } })`
  3. `getAll()` returns `cookieStore.getAll()`
  4. `setAll(cookiesToSet)` iterates and calls `cookieStore.set(name, value, options)` inside a `try/catch` block
  5. The catch block is empty with a comment: "The setAll method was called from a Server Component. This can be ignored if you have middleware refreshing user sessions."

**Critical patterns from research (DO NOT deviate):**
- Use `getAll`/`setAll` cookie methods ONLY. Do NOT use deprecated `get`/`set`/`remove` methods.
- The server `createClient` MUST be `async` because `cookies()` is async in Next.js 14+.
- Do NOT create a singleton for the server client -- each request must get its own instance.
- Do NOT import `cookies()` outside the function body.
- Do NOT use `@supabase/auth-helpers-nextjs` (deprecated).

**Type the `setAll` parameter** using inline typing: `cookiesToSet: { name: string; value: string; options: CookieOptions }[]` with `CookieOptions` imported from `@supabase/ssr` (import type).
  </action>

Verify both files exist and export `createClient`.
  </verify>

<task type="auto">
  <name>Task 2: Create middleware for session refresh</name>
  <files>middleware.ts</files>
  <action>
Create `middleware.ts` at the project root (same level as `app/` directory).

**Implementation:**
- Import `createServerClient` from `@supabase/ssr`
- Import `NextResponse` and `type NextRequest` from `next/server`
- Export an `async function middleware(request: NextRequest)` that:
  1. Creates `let supabaseResponse = NextResponse.next({ request })`
  2. Creates a Supabase server client with `createServerClient(url, anonKey, { cookies: { getAll, setAll } })`
  3. `getAll()` returns `request.cookies.getAll()`
  4. `setAll(cookiesToSet)` does TWO things:
     a. Sets cookies on the request: `cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))`
     b. Creates a new response: `supabaseResponse = NextResponse.next({ request })`
     c. Sets cookies on the response: `cookiesToSet.forEach(({ name, value, options }) => supabaseResponse.cookies.set(name, value, options))`
  5. Calls `await supabase.auth.getUser()` IMMEDIATELY after creating the client (no code between createServerClient and getUser)
  6. Add a comment: "For now, no auth redirects (auth is Phase 4+). This middleware just refreshes the session cookie."
  7. Returns `supabaseResponse`

- Export a `config` object with `matcher` array excluding static files:
  ```
  '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)'
  ```

**Critical patterns (DO NOT deviate):**
- The `supabaseResponse` variable MUST be reassigned inside `setAll` (creating new `NextResponse.next({ request })`) -- this ensures modified request cookies propagate to downstream Server Components.
- `auth.getUser()` MUST be called (not `auth.getSession()`). `getUser()` validates the JWT with the Supabase Auth server. `getSession()` does NOT validate and is insecure on the server.
- NO code between `createServerClient` and `auth.getUser()` call.
- Destructure the result: `const { data: { user } } = await supabase.auth.getUser()` -- the `user` variable is not used yet but the call triggers token refresh.
  </action>
