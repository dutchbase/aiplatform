**Create `app/login/actions.ts`** -- Server Actions for login and signup.

```typescript
'use server'

import { revalidatePath } from 'next/cache'
import { redirect } from 'next/navigation'
import { createClient } from '@/lib/supabase/server'
```

**`login` function:**
- Accept `formData: FormData`
- Create server Supabase client via `await createClient()`
- Extract `email` and `password` from formData using `.get()` and cast as `string`
- Call `supabase.auth.signInWithPassword({ email, password })`
- If `error`, call `redirect('/login?error=invalid-credentials')`
- If success, call `revalidatePath('/', 'layout')` THEN `redirect('/')`
- CRITICAL: Do NOT wrap `redirect()` in try/catch. Next.js `redirect()` works by throwing an internal error. If caught in try/catch, the redirect will silently fail.
- CRITICAL: Call `revalidatePath` BEFORE `redirect`. Otherwise cache invalidation may not take effect.

**`signup` function:**
- Accept `formData: FormData`
- Create server Supabase client via `await createClient()`
- Extract `email`, `password`, and `display_name` from formData
- Call `supabase.auth.signUp({ email, password, options: { data: { display_name } } })`
- The `display_name` in `options.data` lands in `raw_user_meta_data` and is picked up by the database trigger from Plan 01
- If `error`, call `redirect('/login?error=signup-failed')`
- If success, call `revalidatePath('/', 'layout')` THEN `redirect('/login?message=check-email')`
- NOTE: When email confirmation is enabled (Supabase default), `signUp` returns `session: null`. This is NOT an error. Check for `error`, not `session`.

**DO NOT:**
- Use `auth.getSession()` (insecure on server, use `auth.getUser()` when needed)
- Wrap redirect() in try/catch
- Use Zod validation (not needed yet -- Supabase Auth handles email/password validation; Zod comes in later phases)
- Import from `@supabase/auth-helpers-nextjs` (deprecated)

**Create `app/auth/confirm/route.ts`** -- Email confirmation callback for PKCE flow.

This Route Handler handles the redirect when a user clicks the email confirmation link.

```typescript
import { type EmailOtpType } from '@supabase/supabase-js'
import { type NextRequest, NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'
```

Export an `async function GET(request: NextRequest)`:
1. Parse `token_hash` and `type` from `request.url` searchParams
2. Cast `type` as `EmailOtpType | null`
3. Set `const next = '/'` (redirect destination after confirmation)
4. Clone `request.nextUrl` for the redirect URL
5. Set the clone's pathname to `next`
6. Delete `token_hash`, `type`, and `next` from the clone's searchParams (clean URL)
7. If both `token_hash` and `type` exist:
   - Create server client via `await createClient()`
   - Call `supabase.auth.verifyOtp({ type, token_hash })`
   - If no error, redirect to the clean URL via `NextResponse.redirect(redirectTo)`
8. If token_hash/type missing OR verifyOtp errors, redirect to `/error` page
   - Set `redirectTo.pathname = '/error'`
   - Return `NextResponse.redirect(redirectTo)`

**Key details:**
- The token_hash is valid for 5 minutes and can only be used once
- The Supabase email template must be configured to point to this route (see user_setup)
- This handles signup confirmation, password reset, and magic link flows (all use verifyOtp)

**Create `app/auth/signout/route.ts`** -- Sign out route handler.

```typescript
import { createClient } from '@/lib/supabase/server'
import { revalidatePath } from 'next/cache'
import { type NextRequest, NextResponse } from 'next/server'
```

Export an `async function POST(request: NextRequest)`:
1. Create server client via `await createClient()`
2. Call `supabase.auth.getUser()` to check if there is a logged-in user (use `getUser()`, NOT `getSession()`)
3. If user exists, call `await supabase.auth.signOut()`
4. Call `revalidatePath('/', 'layout')` to invalidate cached pages
5. Return `NextResponse.redirect(new URL('/login', request.url), { status: 302 })`

**Key details:**
- Uses POST method (not GET) because signing out is a state mutation
- Checks for user first to avoid unnecessary signOut calls
- Redirects to /login after signout with 302 status
- `revalidatePath('/', 'layout')` ensures all cached Server Components reflect the logged-out state

**DO NOT:**
- Use GET method for signout (POST is correct for mutations)
- Use `auth.getSession()` (insecure on server)
- Skip `revalidatePath` (would leave stale auth state in cached pages)
